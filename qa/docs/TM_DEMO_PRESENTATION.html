The Core Question

"How does the system know which unit test belongs to which business scenario?"

Answer: Pattern-Based Regex Matching

Step 1: QA Defines Scenario with Patterns

// File: qa/matrix/scenario-definitions.ts Scenario: HF001 - "Create user with valid payload" unitTestPatterns: [ 'should create.*user.*success', // Pattern 1 'should create a user', // Pattern 2 'create.*user.*valid' // Pattern 3 ]

What this means:

  • QA Team defines text patterns that identify relevant tests
  • Each pattern is a regex (regular expression)
  • .* means "any characters" - making it flexible
  • One scenario can have multiple patterns for better matching

Step 2: System Parses Unit Tests from Code

// Found in: UserController.test.ts { id: "test_onboarding_user_controller_1", description: "should create a user successfully", // ‚Üê This text service: "onboarding-service", layer: "API" }

4. Orphan Test Management - Detailed

What is an Orphan Test?

A unit test that exists in the codebase but is not mapped to any business scenario. This happens when:

  • Unit test written first: Developer wrote the test, but QA Team hasn't created the corresponding test scenario definition yet
  • Infrastructure test: Test covers technical concerns (DB connection, health checks) - doesn't need business scenario
  • Legacy tests: Unit tests existed before TM system was introduced to the project

üìù Important Note: Development Flow

Typical Order:

  1. Developer writes code + unit tests
  2. QA Team defines test scenarios (business requirements)
  3. TM System maps scenarios ‚Üí unit tests

So yes, unit tests often exist before QA defines the scenario! That's normal in development. The TM system helps bridge this gap by identifying these "orphan" tests.

Feature 1: Smart Classification

System automatically classifies orphans into 2 categories:

Classification What It Means Example Tests Action Needed?
Business Logic Tests that validate business rules, user workflows, API behavior ‚Ä¢ "should validate email format"
‚Ä¢ "should return 400 for duplicate user"
‚Ä¢ "should create order successfully" 		‚ö†Ô∏è YES - QA Action Required
Infrastructure Tests for database connections, config loading, health checks ‚Ä¢ "should connect to MongoDB"
‚Ä¢ "should load environment variables"
‚Ä¢ "should respond to health check" 		‚úÖ NO - No Action Needed

How Classification Works:

  • System analyzes test description keywords
  • Checks test file location (e.g., /infrastructure/ folder)
  • Identifies patterns like "connect", "load", "health", "config"
  • Business logic tests = Need scenario mapping
  • Infrastructure tests = Technical tests, no business scenario needed
